# v1.2 - Authentication & Authorization

What it does:
Enforces user authentication and JWT role-based access control. Users are associated with one of three roles: 'advisor', 'team_leader', or 'director'. Advisor role can only query their own calls and summaries. Team leaders are restricted to their team's scope. Directors have org-wide visibility.

Why built this way (key decision + alternative not picked):
Used direct bcrypt library verification for hashing instead of passlib CryptContext because passlib's bcrypt backend loader has compatibility issues with Python 3.11/3.12 and newer bcrypt package releases. JWT tokens are verified using python-jose, and FastAPI security HTTPBearer checks enforce row-level scoping directly in the route database query layer rather than relying on frontend-only UI filtering.

Inputs / outputs / API contract changes (if any):
- New router endpoints:
  - `POST /auth/login` - Takes `LoginRequest` (email, password), returns access/refresh tokens.
  - `GET /auth/me` - Returns `UserProfileResponse` containing user ID, email, and role.
  - `POST /auth/refresh` - Takes `RefreshRequest` (refresh_token) and returns a new token set.
- Existing endpoints (`/calls`, `/orgs`, `/teams`, `/advisors`, `/tags`, `/contests`) now require a Bearer token in the `Authorization` header and enforce role access restrictions.

Tests added or updated, and why:
- `tests/test_auth.py`: Verifies JWT logins, failed attempts, profile recovery, role enforcement, and row-level access (HTTP 403 blocks advisors from others' data; TLs from other teams' data).
- `tests/test_api.py`: Updated `db_session` fixture to seed a Director account and auto-authorize the test client to keep existing route verification green.

Known gaps:
None. The JWT auth flow and row-level scopes are fully implemented and verified green.
