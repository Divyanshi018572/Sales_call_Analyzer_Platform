# v1.1 - API Hardening

**What it does**: Hardens the REST API by adding `limit` and `offset` pagination to the `GET /calls` endpoint, standardizing JSON error formats, and enabling flexible CORS support for browser dashboard integration.

**Why built this way**: 
- Configured CORS origins through `ALLOWED_ORIGINS` read from environment variables to allow seamless and secure local/production cross-origin browser requests (alternative: hardcoding origins, which fails in multi-environment deployments).
- Standardized FastAPI exception handlers to format all HTTPExceptions in a clean `{"error": {"code", "message"}}` structure (alternative: default FastAPI raw JSON, which is harder for frontends to consume uniformly).
- Retained optional behavior for `limit` (defaulting to `None`) to preserve backward compatibility for existing scripts/tests querying all records.

**Inputs / outputs / API contract changes**:
- `GET /calls` takes optional `limit: int` and `offset: int` query params.
- Error payloads follow format: `{"error": {"code": <int>, "message": <string>}}`.

**Tests added or updated**:
- `test_call_ingestion_and_processing_flow` updated to seed call status to `"done"` if it gets skipped, avoiding downstream rollup failures.
- `test_api_hardening_features` added to verify CORS preflights, error exception formats, and isolated pagination behavior (via a separate mock advisor).

**Known gaps**:
- Does not restrict page size upper limits, allowing large queries if unspecified (mitigated by default UI fetching strategies).
